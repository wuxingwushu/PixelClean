## 详细分析

### O1. 区块顶点增量更新

**现状：** `RebuildAllVertexData()` 在任意区块变化时重建**全部**区块的顶点数据。当前 125 个区块（5×5×5），每个 32³=32768 个方块，每次重建遍历约 4M 方块，对每个非空气方块调用 6 次 `GetBlockAtWorld`（含 mutex 锁）。

**代码位置：** [BlockWorld.cpp](file:///c:/GitHub/PixelClean/GameMods/BlockWorld/BlockWorld.cpp#L285-L305)

```cpp
// 当前：全量重建
for (auto* chunk : chunkSnapshot) {
    BuildChunkVertices(chunk, vertices);  // 所有区块都重建
}
```

**方案 A：Per-Chunk 脏标记 + 独立顶点缓冲**

每个 ChunkData 维护一个 `bool vertexDirty`，`OnChunkGenerate` 设置该 chunk 为脏。`RebuildAllVertexData` 只重建脏区块的顶点，未变化的区块复用已有顶点数据。

- 需要为每个区块维护独立的 Vulkan 顶点缓冲（或使用一个大缓冲 + 偏移）
- 需要为每个区块维护独立的 `vkCmdDraw` 调用
- 优点：极致增量，O(dirty_count) vs O(total_count)
- 缺点：大幅增加 draw call 数量（125 个 vs 1 个），增加 Vulkan 资源管理复杂度，可能反而降低渲染性能
- 预期性能提升：重建耗时降低 80-95%

**方案 B：Per-Chunk 脏标记 + 合并重写单缓冲**

每个 ChunkData 维护 `bool vertexDirty`。重建时扫描所有区块，脏区块重新构建顶点，干净区块从缓存拷贝。所有顶点仍写入同一个大缓冲。

- 需要缓存每个区块的顶点数据（内存开销: 125 × 平均顶点数 × 24 bytes）
- 优点：保持单 draw call，仅重建脏区块
- 缺点：需要额外内存缓存顶点数据（约 5-20MB），实现复杂度中等
- 预期性能提升：重建耗时降低 70-90%

**方案 C：局部脏区合并（相邻区块影响）**

当区块 A 变化时，A 的边界方块会影响相邻区块的面可见性。因此需要标记 A 及其 6 个邻居共 7 个区块为脏。其余区块复用。

- 类似方案 B，但考虑邻居影响
- 优点：渲染正确性更高
- 缺点：如果变化区块多，回退到接近全量重建
- 预期性能提升：单区块变化时 70-85%，多区块同时变化时 30-60%

**推荐：方案 B + 方案 C 的邻居感知**，在正确性和性能间取得最佳平衡。

| 方案 | 性能提升 | 复杂度 | 内存开销 | Draw Call |
|------|----------|--------|----------|-----------|
| A | 80-95% | 高 | 中 | 125× |
| B | 70-90% | 中 | 5-20MB | 1× |
| C | 30-85% | 中 | 5-20MB | 1× |

### O2. 区块数据存储类型优化

**现状：** `ChunkData::blocks` 声明为 `int blocks[32][32][32]`，每个方块 4 字节（32 bits），单个区块 128KB，125 个区块共 **16MB**。

**代码位置：** [ChunkData.h](file:///c:/GitHub/PixelClean/GameMods/BlockWorld/ChunkData.h#L85)

```cpp
int blocks[CHUNK_SIZE][CHUNK_SIZE][CHUNK_SIZE]{};  // 32×32×32×4 = 128KB
```

**方案 A：uint8_t [32][32][32]**

方块类型仅 12 种，用 `uint8_t` 完全够用。8MB 总内存（减半）。

- 优点：内存减半，缓存命中率翻倍，改动极小（改类型 + 改 `get()` 返回值）
- 缺点：未来若超过 256 种方块需再次修改（12 种→256 种，余地充足）
- 预期性能提升：内存 -50%，generateTerrain 速度 +10-15%（更少 cache miss）

**方案 B：uint8_t 一维数组 + 索引宏**

`uint8_t blocks[CHUNK_SIZE * CHUNK_SIZE * CHUNK_SIZE]`，配合 `#define IDX(x,y,z) ((x)*CS*CS + (y)*CS + (z))`。

- 优点：更好的内存局部性，避免 C++ 多维数组的间接访问
- 缺点：代码可读性稍降
- 预期性能提升：比方案 A 多 5-10%（消除多维数组指针解引用）

**方案 C：稀疏存储（Palette + Index）**

使用调色板压缩：统计区块内实际使用的方块类型（通常 3-8 种），用 4-bit 索引替代 8-bit 值。

- 优点：内存降至 4-8MB
- 缺点：编码/解码开销，不适合频繁随机访问
- 预期性能提升：内存 -50% vs 方案 A，但访问速度可能下降 5-10%

**推荐：方案 A**，改动最小，收益直接，风险最低。

| 方案 | 内存 | get() 性能 | 改动量 |
|------|------|-----------|--------|
| A (uint8_t[][][]) | 8MB | 不变 | 极小 |
| B (uint8_t[]) | 8MB | +5% | 小 |
| C (palette) | 4-8MB | -5~10% | 大 |

### O3. GetBlockAtWorld 锁竞争优化

**现状：** `GetBlockAtWorld()` 每次调用都获取 `std::lock_guard<std::mutex>`。在 `BuildChunkVertices` 中，每个非空气方块的 6 个面检查都会调用此函数。对于 125 个区块约 50% 填充率：~125 × 16384 × 6 ≈ **1200 万次 mutex 锁/解锁**。

**代码位置：** [BlockWorld.cpp](file:///c:/GitHub/PixelClean/GameMods/BlockWorld/BlockWorld.cpp#L91-L117)

```cpp
int BlockWorld::GetBlockAtWorld(int wx, int wy, int wz) {
    // ...
    std::lock_guard<std::mutex> lock(mChunkMapMutex);  // 每次调用都上锁
    auto it = mChunkMap.find(std::make_tuple(chunkX, chunkY, chunkZ));
    // ...
}
```

**方案 A：Chunk 邻接指针缓存**

在 `RebuildAllVertexData` 的快照阶段，同时构建一个 chunk 坐标→指针的临时查找表（用 `std::unordered_map` 或直接数组），后续的 `GetBlockAtWorld` 等价逻辑直接从该表无锁查找。

- 优点：完全消除顶点构建期间的 mutex 开销
- 缺点：需要额外在快照阶段构建查找表（O(N) 额外开销，极小）
- 预期性能提升：RebuildAllVertexData 耗时降低 40-60%

**方案 B：std::shared_mutex 读写锁**

将 `std::mutex` 改为 `std::shared_mutex`。`GetBlockAtWorld` 使用 `shared_lock`（读锁，多线程可并发），`OnChunkGenerate` 和 `CleanupOutOfRangeChunks` 使用 `unique_lock`（写锁）。

- 优点：允许多个读操作并发，无需缓存
- 缺点：shared_mutex 开销略大于普通 mutex
- 预期性能提升：RebuildAllVertexData 耗时降低 25-35%（如果有多线程读，实际RebuildAllVertexData是单线程，收益来自减少锁开销，但 mutex 本身仍有开销）

**方案 C：移除 GetBlockAtWorld 中的锁，改为无锁版本（仅用于顶点构建）**

添加 `GetBlockAtWorldUnsafe()` 无锁版本，仅在 `RebuildAllVertexData` 的快照保护下使用。快照已确保所有 chunk 指针在顶点构建期间不会被释放（因为 cleanup 在 vertex dirty 之后且 snapshot 保护了指针）。

- 优点：零锁开销
- 缺点：需要确保调用顺序正确（当前 GameLoop 先 cleanup 后 rebuild，cleanup 已持有锁）
- 预期性能提升：RebuildAllVertexData 耗时降低 50-70%

**推荐：方案 A**，最安全且收益最高。方案 C 收益更大但有并发正确性风险。

| 方案 | 性能提升 | 复杂度 | 风险 |
|------|----------|--------|------|
| A (邻接缓存) | 40-60% | 中 | 低 |
| B (读写锁) | 25-35% | 低 | 低 |
| C (无锁版本) | 50-70% | 低 | 中 |

### O4. 地形生成 Pass 合并

**现状：** `generateTerrain()` 对 32³ 网格执行 **5 次完整遍历**：

| Pass | 遍历范围 | 噪声调用/体素 | 总噪声调用 |
|------|----------|--------------|-----------|
| Pass 1: 密度场+方块类型 | 32³ 全部 | ~9 (2D) + ~3 (3D) | ~400K |
| Pass 2: 洞穴掏空 | 32³ 非Air/Bedrock | ~12-15 | ~200K |
| Pass 3: 含水层 | 32³ Air | ~2 | ~50K |
| Pass 4: 河流水 | 32³ Air | ~2 | ~50K |
| Pass 5: 树木 | (32-6)² 地表 | ~1 | ~700 |

**总计：** ~700K 噪声调用/区块，125 区块 ~87M 噪声调用。重复遍历同一数据导致严重的 cache miss。

**代码位置：** [ChunkData.h](file:///c:/GitHub/PixelClean/GameMods/BlockWorld/ChunkData.h#L218-L417)

**方案 A：单 Pass 合并**

将所有阶段合并为一个 x→y→z 循环，每个体素一次性完成所有计算。

```cpp
for (x) for (y) for (z) {
    // 1. 密度场 → block type
    // 2. 洞穴检测
    // 3. 含水层填充
    // 4. 河流水填充
}
// 5. 树木（仍需单独 pass，因为依赖地表检测）
```

- 优点：数据在 L1/L2 cache 中热乎，消除 4 次完整的 128KB 区块遍历
- 缺点：代码结构不如分 pass 清晰
- 预期性能提升：generateTerrain 耗时降低 25-40%

**方案 B：XYZ 循环顺序优化**

当前循环顺序是 x→y→z（或 x→z→y，在各 pass 间不统一）。确保使用 `x→z→y` 顺序（x 为最内层），因为 `blocks[x][y][z]` 按 x 连续存储，x 最内层循环最大化 cache 行利用。

- 优点：改动极小（仅调整循环顺序）
- 缺点：收益有限，仅消除部分 cache miss
- 预期性能提升：generateTerrain 耗时降低 5-15%

**方案 C：合并 Pass 2-4（洞穴+含水层+河流）**

Pass 1 必须单独运行（因为 Pass 2-4 依赖 Pass 1 的结果）。将 Pass 2-4 合并为一个 y-优先循环。树木 Pass 5 独立。

- 优点：在方案 A 和现状之间折中，代码可读性尚可
- 缺点：仍有多余一次遍历
- 预期性能提升：generateTerrain 耗时降低 15-25%

**推荐：方案 C（短期）+ 方案 A（长期）**。先合并后处理 pass 获得快速收益，后续考虑完整合并。

| 方案 | 性能提升 | 复杂度 | 代码可读性 |
|------|----------|--------|-----------|
| A (全合并) | 25-40% | 高 | 下降 |
| B (循环优化) | 5-15% | 极低 | 不变 |
| C (部分合并) | 15-25% | 中 | 略降 |

### O5. 洞穴检测合并优化

**现状：** 每个非 Air/非 Bedrock 体素依次检测 4 种洞穴类型（cheese, spaghetti, noodle, ravine）。每种类型独立调用噪声，`warpCoordinate` 每种调用 3 次噪声。总计每体素 **12-15 次噪声调用**。

**代码位置：** [ChunkData.h](file:///c:/GitHub/PixelClean/GameMods/BlockWorld/ChunkData.h#L326-L333)

```cpp
if (isCheeseCave(...) || isSpaghettiCave(...) ||
    isNoodleCave(...) || isRavine(...)) {
    blocks[x][y][z] = (int)BlockType::Air;
}
```

**方案 A：早期退出优化**

当前使用 `||` 短路求值，但各函数内部都做完整的 warp + noise 调用。可以在调用前做廉价预检：

- Cheese cave 和 Ravine 有高度范围限制（ravine 只在低处），先检查范围
- Noodle cave 限 wy > 30 时直接跳过
- 利用 `||` 短路：将最可能返回 true 的类型放在前面

- 优点：改动极小，利用现有短路机制
- 缺点：收益取决于实际地形（洞穴多的区域收益低）
- 预期性能提升：Pass 2 耗时降低 10-20%（在洞穴少的区域可达 40%）

**方案 B：统一洞穴噪声函数**

创建一个组合洞穴函数，用单一噪声调用来判定是否属于洞穴。使用高 octave 的 3D Perlin 噪声，通过不同阈值区间对应不同洞穴类型。

- 优点：噪声调用从 12-15 次降至 1 次
- 缺点：洞穴形态可能不如独立噪声丰富，需要仔细调参
- 预期性能提升：Pass 2 耗时降低 60-80%

**方案 C：LOD 降采样检测**

Pass 2 不遍历全部体素：先用 2×2×2 降采样检测每个 2³ 块是否需要掏空。如果降采样结果为全固体，跳过整个 2³ 块；如果有洞穴可能，再逐体素检测。

- 优点：在少洞穴区域大幅减少检测次数（理论减少 87.5% 遍历）
- 缺点：实现复杂度高，可能在洞穴边缘产生伪影
- 预期性能提升：Pass 2 耗时降低 30-50%

**推荐：方案 A（立即实施）+ 方案 C（中期）**。方案 B 风险较高，除非洞穴效果可接受降级。

| 方案 | 性能提升 | 复杂度 | 视觉影响 |
|------|----------|--------|----------|
| A (早期退出) | 10-40% | 极低 | 无 |
| B (统一噪声) | 60-80% | 中 | 有（洞穴多样性降低） |
| C (LOD) | 30-50% | 高 | 边缘伪影风险 |

### O6. ChunkKey 哈希优化

**现状：** `ChunkKey` 使用 `std::tuple<int, int, int>`，每次查找需构造临时 tuple 对象，哈希函数调用 3 次 `std::get<>()`。

**代码位置：** [BlockWorld.h](file:///c:/GitHub/PixelClean/GameMods/BlockWorld/BlockWorld.h#L29-L35)

```cpp
struct ChunkKeyHash {
    size_t operator()(const std::tuple<int, int, int>& key) const {
        return std::get<0>(key) * 73856093
             ^ std::get<1>(key) * 19349663
             ^ std::get<2>(key) * 83492791;
    }
};
```

**方案 A：打包 int64 key**

chunk 坐标范围有限（-32768 ~ 32767 用 int16 足够），可打包为单个 `int64_t`：

```cpp
using ChunkKey = int64_t;
inline ChunkKey MakeKey(int x, int y, int z) {
    return ((int64_t)(x & 0xFFFF) << 32)
         | ((int64_t)(y & 0xFFFF) << 16)
         |  (int64_t)(z & 0xFFFF);
}
// 无需自定义 hash，直接使用 std::hash<int64_t>
```

- 优点：零开销 hash（int64 本身即 key），无 tuple 构造，find() 更快
- 缺点：坐标范围限制（但 16-bit 足够）
- 预期性能提升：chunk 查找耗时降低 30-50%

**方案 B：使用结构体 + 内联方法**

```cpp
struct ChunkKey {
    int x, y, z;
    bool operator==(const ChunkKey& o) const { return x==o.x && y==o.y && z==o.z; }
};
struct ChunkKeyHash {
    size_t operator()(const ChunkKey& k) const {
        return ((size_t)k.x * 73856093) ^ ((size_t)k.y * 19349663) ^ ((size_t)k.z * 83492791);
    }
};
```

- 优点：避免 `std::tuple` 和 `std::get<>()` 的模板开销
- 缺点：仍需要自定义 hash，但比 tuple 快
- 预期性能提升：chunk 查找耗时降低 15-25%

**推荐：方案 A**，极致简洁高效。

| 方案 | 性能提升 | 复杂度 | 坐标限制 |
|------|----------|--------|----------|
| A (int64 打包) | 30-50% | 极低 | ±32767 |
| B (struct) | 15-25% | 极低 | 无 |

### O7. 地形生成噪声调用缓存

**现状：** Pass 1 中每个 (x,z) 列调用 6 种 2D 噪声（continental, erosion, temperature, humidity, weirdness, ridge），每个 (x,y,z) 体素调用 3 种 3D 噪声（densityNoise1/2/3）。这些 2D 噪声对同一 (x,z) 列的所有 y 层重复计算。

**代码位置：** [ChunkData.h](file:///c:/GitHub/PixelClean/GameMods/BlockWorld/ChunkData.h#L229-L311)

**已有缓存：** `biomeCache[x][z]` 和 `erodedHeightCache[x][z]` 已缓存了部分结果。但 6 种 2D 噪声原始值和 3 种 density 噪声未缓存。

**方案 A：全量 2D 噪声预计算**

在 Pass 1 开始前，对 32×32=1024 个 (x,z) 位置预计算所有 2D 噪声（continental, erosion, temperature, humidity, weirdness, ridge），存入 `float[32][32]` 数组。

- 优点：消除 32 × 6 × 32 = 6144 次冗余噪声调用
- 缺点：额外 32×32×6×4 ≈ 24KB 栈空间
- 预期性能提升：Pass 1 耗时降低 5-10%

**方案 B：2D + 3D 全预计算**

除方案 A 外，对每个 (x,z) 列的每个 y 层预计算 density 噪声。对于 32³ 全网格，需 32³×3×4 ≈ 384KB。可分批计算以控制内存。

- 优点：完全消除噪声调用的交叉依赖，便于 SIMD 批处理
- 缺点：内存开销大
- 预期性能提升：Pass 1 耗时降低 10-20%

**方案 C：利用 FastNoiseLite 的 Fractal 噪声替代多重噪声**

当前 densityNoise1/2/3 分别用不同 octave/frequency 采样。可用一个高 octave 的单一 Fractal 噪声替代，减少调用次数。

- 优点：3 次噪声调用→1 次
- 缺点：密度场精度可能下降
- 预期性能提升：Pass 1 耗时降低 5-10%

**推荐：方案 A**，简单直接，栈开销可接受。

| 方案 | 性能提升 | 内存开销 | 精度影响 |
|------|----------|----------|----------|
| A (2D 预计算) | 5-10% | 24KB 栈 | 无 |
| B (全预计算) | 10-20% | 384KB | 无 |
| C (合并噪声) | 5-10% | 0 | 有 |

### O8. 顶点缓冲区扩容策略优化

**现状：** 当顶点数超过 `mVertexCapacity`（初始 10M），触发 `DestroyBlockWorldVulkanResources()` 释放所有 Vulkan 资源，然后重新创建 Buffer、DescriptorPool、DescriptorSet、CommandPool、CommandBuffers。这会导致一到多帧的画面闪烁/卡顿。

**代码位置：** [BlockWorld.cpp](file:///c:/GitHub/PixelClean/GameMods/BlockWorld/BlockWorld.cpp#L316-L353)

**方案 A：更大初始容量**

将 `mVertexCapacity` 从 `10000000` 提升到 `50000000`（50M），覆盖更大范围的地形。125 个区块满额约 20M 顶点，50M 提供 2.5x 余量。

- 优点：一行修改，几乎不会触发扩容
- 缺点：初始即分配 50M×24B = 1.2GB 顶点缓冲（取决于 Vertex 结构体大小）
- 预期效果：扩容频率降低 80-90%

**方案 B：双缓冲 + 异步重建**

维护两个顶点缓冲（front/back）。渲染用 front，重建写入 back，完成后原子交换。扩容时只重建 back buffer，不阻塞渲染。

- 优点：零帧闪烁，扩容对用户透明
- 缺点：内存翻倍，逻辑复杂度增加
- 预期效果：消除扩容导致的画面卡顿

**方案 C：只重建 Buffer，复用其他资源**

扩容时仅重建 `mVertexBuffer`，不销毁 DescriptorSet/CommandPool 等。因为 DescriptorSet 绑定的是 Buffer 对象（VkBuffer handle），而非 Buffer 大小。重建 Buffer 后重新绑定即可。

- 优点：改动小，扩容代价从完整重建→单个 Buffer 重建
- 缺点：需要验证 DescriptorSet 是否依赖 Buffer 大小（通常不依赖）
- 预期效果：扩容耗时降低 70-80%

**推荐：方案 C + 方案 A 结合**。先修改扩容逻辑为最小重建，再适当提升初始容量。

| 方案 | 扩容代价 | 内存开销 | 复杂度 |
|------|----------|----------|--------|
| A (更大容量) | 不变但频率低 | +400MB | 极低 |
| B (双缓冲) | 0 卡顿 | 2× | 高 |
| C (最小重建) | -70% | 不变 | 低 |

### O9. 相机方向向量缓存

**现状：** `KeyDown()` 每帧被调用多次（持续按键），每次都重新计算 `cos(pitch)*sin(yaw)`、`sin(pitch)`、`cos(pitch)*cos(yaw)` 和 `normalize(cross(...))`。

**代码位置：** [BlockWorld.cpp](file:///c:/GitHub/PixelClean/GameMods/BlockWorld/BlockWorld.cpp#L760-L772)

```cpp
glm::vec3 forward(
    cos(pitchRad) * sin(yawRad),
    sin(pitchRad),
    cos(pitchRad) * cos(yawRad)
);
glm::vec3 right = glm::normalize(glm::cross(worldUp, forward));
```

**方案 A：缓存 forward/right**

在 `UpdateCamera()` 或 `MouseMove()` 中计算并缓存 `mCameraForward` 和 `mCameraRight`。`KeyDown()` 直接使用缓存值。

- 优点：消除每帧的三角运算和 cross + normalize
- 缺点：需确保缓存更新时机正确（每次 yaw/pitch 变化后）
- 预期性能提升：KeyDown 耗时降低 60-80%（但 KeyDown 本身开销极低，整体帧时间影响 <0.1%）

**推荐：方案 A**，改动极小，一次性修复。

### O10. 多线程任务同步机制优化

**现状：** `OnChunkGenerate` 使用双重检查锁定模式（DCLP），两次 `find()` 和两次 `lock_guard`。`MovePlate3D::UpData` 中每发现一个新区块就 fire-and-forget 到线程池，不等待任务完成。`GameLoop` 立即检查 `mVertexDataDirty`，但此时异步任务可能还未完成。

**代码位置：** [BlockWorld.cpp](file:///c:/GitHub/PixelClean/GameMods/BlockWorld/BlockWorld.cpp#L18-L80)

**方案 A：收集 future 并等待**

修改 `MovePlate3D::UpData`（或新增方法）收集 `enqueue` 返回的 `std::future<void>`，在返回前等待所有生成任务完成。

- 优点：GameLoop 中的 RebuildAllVertexData 看到的是完整数据
- 缺点：主线程被阻塞，失去了异步生成的优势
- 预期效果：帧率可能下降（等待生成），但数据一致性更好

**方案 B：延迟一帧处理**

`mVertexDataDirty` 设置时不立即重建，而是在下一帧（或固定延迟 N 帧后）再重建。给异步任务足够时间完成。

- 优点：保留异步优势，让用户感知延迟而非卡顿
- 缺点：新生成的区块需要 1-2 帧才能渲染（视觉上几乎不可见）
- 预期效果：避免竞态条件的同时保持帧率

**方案 C：原子计数器 + 回调**

添加 `std::atomic<int> mPendingGenerates`。每次 enqueue 前 +1，生成完成时在回调中 -1。`GameLoop` 只在 `mPendingGenerates == 0` 且 `mVertexDataDirty` 时才重建。

- 优点：精确知道所有异步任务完成的时机
- 缺点：需要修改回调机制
- 预期效果：完美同步，帧率影响最小

**推荐：方案 B**，实现简单，用户体验无感知。

| 方案 | 帧率影响 | 数据一致性 | 复杂度 |
|------|----------|-----------|--------|
| A (等 future) | 下降 | 完美 | 中 |
| B (延迟一帧) | 无 | 高 | 低 |
| C (原子计数) | 几乎无 | 完美 | 中 |

### O11. Spline 查找二分优化

**现状：** `splineInterpolate()` 使用线性扫描查找 x 所在的区间。9 个点，平均扫描 4.5 次。

**代码位置：** [ChunkData.h](file:///c:/GitHub/PixelClean/GameMods/BlockWorld/ChunkData.h#L46-L54)

```cpp
size_t i = 0;
while (i < points.size() - 1 && points[i + 1].input < x) i++;
```

**方案 A：std::lower_bound 二分查找**

```cpp
auto it = std::lower_bound(points.begin(), points.end(), x,
    [](const SplinePoint& p, float v) { return p.input < v; });
```

- 优点：O(log n) vs O(n)，4 次比较 vs 4.5 次（9 点差异极小）
- 缺点：迭代器开销稍高
- 预期性能提升：splineInterpolate 耗时降低 5-10%（但此函数在总耗时中占比 <0.01%）

**方案 B：硬编码 if-else**

9 个点，直接写 if-else 展开，无循环无查找。

- 优点：极致快，无分支预测失败
- 缺点：代码僵化，修改 spline 点需要改代码
- 预期性能提升：splineInterpolate 耗时降低 20-30%

**推荐：暂不优化**（此函数在整体性能中占比极低，不值得投入）。如要优化选方案 B。

### O12. 噪声实例管理 RAII 化

**现状：** 20 个 `FastNoiseLite*` 成员变量，在 `InitBlockWorldPipeline()` 中逐个 `new`，在 `DestroyBlockWorldPipeline()` 中逐个 `delete`。销毁顺序错误会导致悬挂指针（当前已确保顺序，但容易出错）。

**代码位置：** [BlockWorld.cpp](file:///c:/GitHub/PixelClean/GameMods/BlockWorld/BlockWorld.cpp#L371-L549)

**方案 A：std::unique_ptr 替代裸指针**

将 20 个成员从 `FastNoiseLite*` 改为 `std::unique_ptr<FastNoiseLite>`。Init 中 `std::make_unique`，Destroy 简化为 `reset()` 或自动析构。

- 优点：零泄漏风险，DestoryBlockWorldPipeline 从 80 行缩减到 5 行
- 缺点：头文件需要 `#include <memory>`
- 预期性能提升：无（运行时不影响），但代码质量显著提升

**方案 B：噪声结构体封装**

```cpp
struct NoiseSet {
    std::unique_ptr<FastNoiseLite> continental, erosion, detail;
    std::unique_ptr<FastNoiseLite> density1, density2, density3, ridge;
    std::unique_ptr<FastNoiseLite> temperature, humidity, weirdness;
    std::unique_ptr<FastNoiseLite> cheese, spaghetti, noodle, warpX, warpY, warpZ, ravine;
    std::unique_ptr<FastNoiseLite> aquifer, aquiferBarrier, river;
};
```

- 优点：一个成员变量替代 20 个，代码更整洁
- 缺点：需要调整所有噪声访问点
- 预期性能提升：无，代码质量提升

**推荐：方案 A**，渐进式改进。
